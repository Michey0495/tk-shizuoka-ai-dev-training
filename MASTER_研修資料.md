### 1-4. 生成AI開発トレンド概観 [4min]

#### 生成AI開発トレンドとは

概要:
生成AI（Generative AI）を活用したソフトウェア開発手法の総称。コード生成、テスト自動化、ドキュメント作成、設計支援など、開発ライフサイクル全体にAIを統合する動きを指す。2022年のGitHub Copilot一般公開、2023年のGPT-4登場を経て、2024-2026年にかけて急速に成熟期に入った。

沿革:
- 2021年: GitHub Copilot テクニカルプレビュー開始
- 2022年: GitHub Copilot 一般公開、Tabnine Pro リリース
- 2023年: GPT-4登場、Claude 2リリース、Cursor登場
- 2024年: Claude 3シリーズ、GPT-4o、Gemini 1.5 Pro登場
- 2025年: Claude Code、GitHub Copilot Agent Mode、Kiro登場
- 2026年: 各ツールの成熟と統合が進む

特徴:
- コード生成の精度が飛躍的に向上
- マルチモーダル対応（画像、音声、動画の理解）
- エージェント型AI（自律的にタスクを遂行）
- IDE統合からCLI統合へ拡大

メリット:
- 開発速度の大幅向上（一部タスクで10倍以上）
- ボイラープレートコードの削減
- 学習コストの低減（AIが教師役）
- コードレビューの効率化

デメリット:
- AIへの過度な依存リスク
- 生成コードのセキュリティ問題
- ライセンス・著作権の曖昧さ
- 組織のガバナンス整備が追いつかない

活用事例:
- Shopify: 開発者の50%以上がGitHub Copilotを常用
- Stripe: ドキュメント生成にAIを活用し、更新頻度を3倍に
- メルカリ: 社内AIコーディングアシスタントを開発・導入

競合・類似概念:
| 用語 | 説明 |
|------|------|
| AI駆動開発 | AIが開発プロセスの中核を担う開発手法 |
| バイブコーディング | 自然言語でコードを書く開発スタイル |
| エージェントコーディング | AIエージェントが自律的にコードを生成 |
| ペアプログラミング2.0 | AIをペアとして開発する手法 |

URL:

### 1-5. 開発における注意点 [3min]

#### リスク・ガバナンスとは

概要:
生成AIを開発に導入する際に考慮すべきリスクと、それを管理するための組織的な仕組み。コード品質、セキュリティ、ライセンス、プライバシーなど多岐にわたる観点が含まれる。

主要なリスク領域:

1. セキュリティリスク
   - 生成コードに脆弱性が含まれる可能性
   - 機密情報のAIへの送信
   - サプライチェーン攻撃の潜在リスク

2. ライセンス・著作権リスク
   - 学習データの著作権問題
   - 生成コードの権利帰属
   - OSSライセンス違反の可能性

3. 品質リスク
   - AIが生成したコードのレビュー不足
   - 動作するが最適でないコード
   - テスト不足による本番障害

4. 依存リスク
   - エンジニアのスキル低下
   - ベンダーロックイン
   - サービス停止時の影響

ガバナンス施策の例:
- AIツール利用ポリシーの策定
- 機密情報のマスキングルール
- 生成コードのレビュー必須化
- 定期的なセキュリティ監査
- 利用ログの取得と分析

活用事例:
- 金融機関A社: AI生成コードは必ず2名以上でレビュー
- 製造業B社: 社内プロキシ経由でAIサービスを利用し、ログを全て保存
- IT企業C社: AIツール利用ガイドラインを全社展開

URL:
- OWASP LLM Top 10: https://owasp.org/www-project-top-10-for-large-language-model-applications/
- GitHub Copilot Trust Center: https://resources.github.com/copilot-trust-center/
- NIST AI Risk Management Framework: https://www.nist.gov/itl/ai-risk-management-framework

### 2-3. Agentモード [5min]

#### Agent Modeとは

概要:
GitHub Copilotの新機能。単純なコード補完を超え、タスクを理解して複数のファイルを横断した変更を自律的に行う。開発者は高レベルな指示を出し、Copilotがその実現方法を考えて実行する。

沿革:
- 2024年末: プレビュー発表
- 2025年3月: パブリックプレビュー
- 2025年後半: 一般提供開始
- 2026年: 安定版として広く普及

特徴:
- 自然言語でタスクを依頼
- 複数ファイルの同時編集
- ターミナルコマンドの実行
- エラー修正の自動試行
- 作業のプレビューと承認

メリット:
- 反復的な作業の自動化
- 大規模リファクタリングの効率化
- 複雑なタスクを分解して実行
- 開発者は「何を作るか」に集中できる

デメリット:
- 実行内容の確認が必要
- 想定外の変更が発生する可能性
- コスト増加（トークン消費）
- まだ発展途上

活用事例:
- テスト追加: 「このファイルにユニットテストを追加して」
- リファクタリング: 「このクラスをより小さな関数に分割して」
- 機能追加: 「ユーザー認証機能を追加して」

URL:

### 2-4. フォルダ構成とRules [5min]

#### プロジェクトフォルダ構成のベストプラクティス

AIが理解しやすいフォルダ構成にすることで、生成コードの品質が向上する。

#### Copilot Instructionsとは

概要:
プロジェクト固有のルールや指示をCopilotに伝えるための仕組み。`.github/copilot-instructions.md`に記述すると、すべての提案に反映される。

特徴:
- Markdown形式で記述
- プロジェクトルートに配置
- チームで共有可能
- バージョン管理対象

例:
```markdown
# Copilot Instructions

## コーディング規約
- TypeScriptを使用し、any型は禁止
- 関数は純粋関数として実装
- エラーはResult型でハンドリング

## プロジェクト固有ルール
- APIクライアントは src/api/client.ts を使用
- 日付処理は dayjs を使用（Dateオブジェクト直接使用禁止）
- ログ出力は src/utils/logger.ts を使用

## 禁止事項
- console.log を本番コードに残さない
- 同期的なファイルI/Oを使用しない
```

### 2-5. AI並列駆動開発 [5min]

#### Git Worktreeとは

概要:
Gitの機能で、1つのリポジトリから複数の作業ディレクトリを同時に作成できる。ブランチごとに別のフォルダで作業でき、切り替えなしに並行開発が可能。AI駆動開発では、複数のAIセッションを同時に走らせる基盤として活用される。

沿革:
- 2015年: Git 2.5でworktree機能追加
- 2020年以降: リモートワーク増加で注目
- 2024年以降: AI並列開発で再評価

特徴:
- 1リポジトリ、複数作業ディレクトリ
- ブランチ切り替え不要
- 各worktreeは独立
- .gitは共有

メリット:
- コンテキストスイッチ削減
- 複数機能の同時開発
- ビルド時間の有効活用
- AIエージェントの並列実行

デメリット:
- ディスク容量を消費
- 管理が複雑になる可能性
- チームへの教育コスト

コマンド例:
```bash
# 新しいworktreeを作成
git worktree add ../feature-auth feature/auth

# worktree一覧を確認
git worktree list

# worktreeを削除
git worktree remove ../feature-auth
```

URL:
- Git Worktree Docs: https://git-scm.com/docs/git-worktree
- Atlassian Git Worktree: https://www.atlassian.com/git/tutorials/git-worktree

---

#### BOSS-Worker構成とは

概要:
AI並列駆動開発のアーキテクチャパターン。1つのBOSS（監督AI）が全体を統括し、複数のWorker（作業AI）に具体的なタスクを割り振る。最終的に評価AI（Reviewer）が成果物をチェックする。

構成:
```
                    ┌─────────────┐
                    │   BOSS AI   │
                    │ (統括・計画) │
                    └──────┬──────┘
                           │
          ┌────────────────┼────────────────┐
          ▼                ▼                ▼
   ┌────────────┐   ┌────────────┐   ┌────────────┐
   │ Worker AI  │   │ Worker AI  │   │ Worker AI  │
   │ (機能A実装) │   │ (機能B実装) │   │ (テスト作成)│
   └────────────┘   └────────────┘   └────────────┘
          │                │                │
          └────────────────┼────────────────┘
                           ▼
                    ┌─────────────┐
                    │ 評価 AI     │
                    │ (レビュー)   │
                    └─────────────┘
```

特徴:
- タスクの分割と並列実行
- 各AIが専門領域に集中
- 人間は最終承認に集中

メリット:
- 開発速度の大幅向上
- 専門性の高い出力
- 人間の負荷軽減

デメリット:
- オーケストレーションが複雑
- コストが増加
- 整合性の担保が課題

活用事例:
- 新機能開発: BOSS がタスク分解、Worker が実装、評価AIがPRレビュー
- リファクタリング: 各Worker が担当モジュールを並列修正

#### Rulesの重要性

すべてのAI搭載IDEに共通する重要概念が「Rules（ルール）」。AIにプロジェクト固有の指示を与え、一貫した出力を得るための仕組み。

ツール別のRulesファイル:
| ツール | ファイル名 | 配置場所 |
|--------|-----------|---------|
| Cursor | .cursorrules | プロジェクトルート |
| Copilot | copilot-instructions.md | .github/ |
| Kiro | .kirorc | プロジェクトルート |

Rules設計のポイント:
- 技術スタック（言語、フレームワーク、バージョン）
- コーディング規約（命名規則、フォーマット）
- アーキテクチャ制約（依存関係、レイヤー構造）
- 禁止事項（使ってはいけないライブラリ等）
- セキュリティ要件（認証、暗号化方式）

#### コンテキストの扱い方

コンテキストの種類:

効果的なコンテキスト管理:
- 必要なファイルだけを渡す（ノイズ削減）
- 関連ドキュメントを明示的に参照
- 会話履歴を適切にリセット
- プロジェクト設定ファイルの活用

```markdown
# プロジェクト概要

- Python 3.11
- FastAPI
- PostgreSQL
- Redis

# 重要ファイル
- src/main.py: エントリーポイント
- src/api/routes/: APIルーティング
- src/models/: データモデル

# 規約
- 関数名はsnake_case
- 型ヒント必須
```

- ツール間の相互運用性
- セキュリティ考慮済み設計
- クライアント/サーバー型アーキテクチャ

アーキテクチャ:
```
┌─────────────┐     MCP     ┌─────────────────┐
│   AI IDE    │ ◀────────▶  │  MCP Server     │
│ (Client)    │             │  (ファイル/DB等) │
└─────────────┘             └─────────────────┘
                                    │
                                    ▼
                            ┌─────────────────┐
                            │   データソース   │
                            │ (Drive, Notion等)│
                            └─────────────────┘
```

メリット:
- プラグインのようにデータソースを追加

- コミュニティ製コネクタが豊富

主要なMCPサーバー:

|---------|------|

| Slack | メッセージ、チャンネル |

### 4-1. 準備されたディスカッション題材

#### Q1: AIの進化による技術の均一化と成長機会の損失について

背景:
AIがコードを書く時代において、エンジニア間のスキル差が縮まりつつある。かつては「書けるかどうか」が差別化要因だったが、AIの支援により誰でも一定品質のコードが書けるようになった。

議論のポイント:
- 「書く力」から何が差別化要因になるか
- AIに任せられない領域とは
- 手を動かす経験の価値
- 失敗から学ぶ機会の確保

考えられる回答:
1. 差別化要因のシフト
   - 要件定義力（何を作るべきかを見極める）
   - アーキテクチャ設計力（全体最適を考える）
   - ドメイン知識（業界特有の知見）
   - コミュニケーション力（ステークホルダーとの対話）

2. AIに任せられない領域
   - 倫理的判断
   - 顧客との信頼関係構築
   - 組織の意思決定
   - 新規性の高いイノベーション

3. 成長機会の再設計
   - AIの出力を評価・改善する訓練
   - 複雑な問題を分解する練習
   - AIがない環境での基礎訓練期間の確保
   - メンタリングによる暗黙知の伝承

#### Q2: ジュニア/ミドル層エンジニアの育成とシニアエンジニアの行動変容

背景:
AIがジュニアエンジニアの「修行」を肩代わりすることで、基礎スキルの習得機会が減少している懸念がある。シニアエンジニアはこの変化にどう対応すべきか。

議論のポイント:
- ジュニアに何を経験させるべきか
- シニアの役割はどう変わるか
- 評価基準の見直し
- チーム構成への影響

考えられる回答:

1. ジュニア育成のアプローチ変化

   従来:
   ```
   簡単なタスク → 徐々に複雑なタスク → 一人前
   ```

   今後:
   ```
   AI活用の型を学ぶ → AIの出力を評価する →
   AIが苦手な領域を担当 → AIを監督する立場へ
   ```

2. シニアエンジニアが変えるべき行動
   - コードレビューの観点変更（AI生成前提）
   - 「教える」から「導く」へ（答えを与えない）
   - AIの限界を示すケーススタディ共有
   - 設計・判断の「なぜ」を言語化する習慣

3. 評価基準の見直し
   - 生産量 → 判断の質
   - コード量 → 問題解決力
   - 個人成果 → チーム貢献

4. 推奨されるプラクティス
   - 週1回の「AIなし開発タイム」
   - AIの出力を3回は修正してからマージ
   - 障害対応はジュニアにも参加させる
   - 技術選定の議論にジュニアを巻き込む
