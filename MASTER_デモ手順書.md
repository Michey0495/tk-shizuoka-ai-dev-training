# Session02 デモ: ブラウザ用AIチャットアプリ開発

## 目的
VSCodeとGitHub Copilotを使った「バイブコーディング」を実演し、自然言語からアプリケーションを生成する体験を共有する。

## 所要時間
25分（説明込み）

## 講師が見せるべきポイント
1. プロンプト一発でアプリの骨格が生成される様子
2. Agentモードによる自律的なコード生成
3. Git Worktreeを使った並列作業の効率性
4. 人間の役割は「指示」と「判断」であること

---

## Part A: バイブコーディングでアプリ生成 [10分]

### 準備状態
- VSCodeで空のフォルダを開いている
- GitHub Copilot Chatが有効
- ターミナルが表示されている

### Step 1: デモの導入（説明）

講師トーク例:
```
これから「バイブコーディング」をお見せします。
バイブコーディングとは、自然言語で作りたいものを伝え、
AIにコードを書いてもらうスタイルです。

今日は「営業担当がお客様との会話メモを貼り付けるだけで、
エンジニアへの引継ぎに必要な技術要件を自動抽出するアプリ」
を作ります。
```

### Step 2: Copilot Chatを開く

1. `Cmd + Shift + I`（Mac）または `Ctrl + Shift + I`（Windows）でCopilot Chatを開く
2. 参加者に画面が見えていることを確認

### Step 3: プロンプトを入力

以下のプロンプトをコピー＆ペーストして実行。

```
あなたは優秀なアプリ開発者です。

営業担当がお客様との会話メモを貼り付けるだけで、
エンジニアへの引継ぎに必要な「技術要件」を自動で抽出・整理してくれるWebアプリを作ってください。

技術要件:
- HTML/CSS/JavaScriptのみ（フレームワーク不要）
- 1ファイル完結（index.html）
- レスポンシブデザイン
- ローカルで動作（サーバー不要）
- OpenAI APIを使用してテキスト解析

機能:
1. テキストエリアに会話メモを入力
2. 「解析」ボタンで技術要件を抽出
3. 抽出結果を以下のカテゴリで表示:
   - 必須機能
   - 希望機能
   - 技術的制約
   - スケジュール要件
   - 予算感
4. 結果をクリップボードにコピーする機能

UIデザイン:
- モダンで清潔感のあるデザイン
- 入力エリアと結果エリアを左右に配置
- ローディング表示あり
```

### Step 4: 生成過程を解説

生成中に参加者へ説明:
```
今、Copilotがコードを生成しています。
注目してほしいのは、私が1行もコードを書いていないこと。
自然言語で要件を伝えるだけで、構造化されたコードが出てきます。

これがバイブコーディングの本質です。
エンジニアの仕事が「コードを書く」から「何を作るか考える」にシフトしています。
```

### Step 5: 生成されたコードを適用

1. 「Apply」または「Insert」ボタンをクリック
2. `index.html` として保存

### Step 6: 動作確認

1. ファイルをブラウザで開く（右クリック → Open with Live Server、または直接開く）
2. サンプルの会話メモを入力してデモ

サンプル会話メモ:
```
2026/2/10 A社 田中様との打ち合わせ

・現在の基幹システムが古くなってきた
・Excelで管理している在庫データを一元化したい
・スマホからも見れると嬉しい
・予算は500万くらいを想定
・来年度の4月から使い始めたい
・セキュリティは重要、社外からのアクセスはNG
・できれば既存のERPと連携したい
```

### Step 7: 結果を見せながら解説

```
ご覧のように、散文的な会話メモから、
エンジニアが必要とする情報が構造化されて出てきました。

必須機能、希望機能、制約条件、スケジュール、予算。
これを手作業でやると30分、AIなら3秒です。

しかも、このアプリ自体を作るのにかかった時間は5分。
バイブコーディングの威力がお分かりいただけたでしょうか。
```

## Part C: Git Worktreeを用いた並列開発 [10分]

### 目的
複数のAIタスクを並列実行し、開発効率を劇的に向上させる手法を見せる

### Step 1: Git Worktreeの説明

講師トーク例:
```
さて、ここからは上級テクニックです。
「Git Worktree」を使った並列開発をお見せします。

通常、Gitリポジトリは1つの作業ディレクトリに1つのブランチ。
ブランチを切り替えるにはcheckoutが必要です。

Git Worktreeを使うと、1つのリポジトリに
複数の作業ディレクトリを持てます。
つまり、複数の機能を同時に開発できるのです。
```

### Step 2: Worktreeの作成（デモ）

ターミナルで以下を実行:

```bash
# 現在のディレクトリを確認
pwd

# Worktreeを2つ作成
git worktree add ../feature-auth feature/authentication
git worktree add ../feature-dashboard feature/dashboard

# Worktree一覧を確認
git worktree list
```

講師コメント:
```
今、3つの作業ディレクトリができました。
- メイン: 本体
- feature-auth: 認証機能の開発
- feature-dashboard: ダッシュボード機能の開発

これらは独立しているので、同時に別々のAIタスクを実行できます。
```

### Step 3: 複数VSCodeウィンドウでの並列作業

1. 3つのフォルダをそれぞれ別のVSCodeウィンドウで開く
2. それぞれのウィンドウでCopilot Chatを起動
3. 各ウィンドウに異なるプロンプトを投入

Window 1（メイン）:
```
このアプリにユーザーテストのシナリオを作成してください。
5つのテストケースをtest/scenarios.mdに出力。
```

Window 2（feature-auth）:
```
このアプリに簡易的なログイン機能を追加してください。
LocalStorageでユーザー情報を管理。
ログイン画面、ユーザー登録画面を作成。
```

Window 3（feature-dashboard）:
```
このアプリに管理者向けダッシュボードを追加してください。
- 解析回数の統計
- よく出現するキーワードの一覧
- 使用状況のグラフ
```

### Step 4: 並列実行の威力を見せる

講師コメント:
```
3つのAIが同時に動いています。
1つのタスクを待っている間に、別のタスクが進む。
シーケンシャルにやれば30分かかる作業が、10分で終わります。

これが「BOSS, Worker, 評価AI」のパターンです。
BOSSであるエンジニアが指示を出し、
WorkerであるAIが並列で実装し、
評価AIまたは人間がレビューする。

開発のボトルネックが「コードを書く時間」から
「何を作るか考える時間」に変わるのです。
```

### Step 5: マージのデモ（時間があれば）

```bash
# feature-authの変更をmainにマージ
cd ../main
git merge feature/authentication

# Worktreeを削除
git worktree remove ../feature-auth
```

## 想定される質問と回答

### Q1: 生成されたコードの品質は大丈夫？

A: 「品質管理は必須です。AIが生成したコードを盲信するのは危険。必ず以下を行ってください。」
- コードレビュー（人間またはAI）
- 自動テストの実行
- セキュリティスキャン
- 本番環境へのデプロイ前に検証

「AIはジュニアエンジニアと同じ。できる仕事は多いが、監督は必要です。」

### Q2: 機密情報がAIに送信されるのでは？

A: 「重要な懸念点です。以下の対策を推奨します。」
- GitHub Copilot Business/Enterpriseは学習に使用されない設定が可能
- 機密情報を含むファイルは.gitignoreに追加
- 社内ポリシーでAIに送信してよい情報を明確化
- ローカルLLM（Ollama等）の検討

### Q3: すべてのエンジニアがこれを使うべき？

A: 「スキルレベルと業務内容によります。」
- シニア: アーキテクチャ設計、コードレビュー、AIの指示に注力
- ミドル: AIと協働、生成コードの品質管理
- ジュニア: AI活用で学習を加速、基礎は別途学ぶ必要あり

「AIはツール。使いこなす人が成果を出せます。」

### Q4: GitHubなしでも使える？

A: 「選択肢はいくつかあります。」
- GitHub Copilot: GitHubアカウント必須
- Cursor: 独立したエディタ、GitHubなしでOK
- Claude Code: CLI、Anthropicアカウントで利用可能
- ローカルLLM: Ollama + Continue等で完全オフライン

## Part A: MCPとは何か [10分]

### Step 1: MCPの説明

講師トーク例:
```
MCPは「Model Context Protocol」の略です。
Anthropic社が提唱したオープンプロトコルで、
AIモデルと外部データソースを接続する標準規格です。

通常、AIはトレーニング時のデータしか知りません。
あなたの会社のドキュメント、DB、APIは知らない。

MCPを使うと、AIがリアルタイムで外部データを参照できます。
「社内Wikiを検索して」「DBから顧客一覧を取得して」
といった指示が可能になるのです。
```

### Step 2: 図解で説明（スライドまたはホワイトボード）

```
┌─────────────┐
│   AI Model  │
└──────┬──────┘
       │
┌──────▼──────┐
│     MCP     │ ← 標準プロトコル
└──────┬──────┘
       │
┌──────▼──────┐
│ MCP Server  │ ← 各種データソースとの橋渡し
└──────┬──────┘
       │
  ┌────┴────┐
  ▼         ▼
┌─────┐  ┌─────┐
│ DB  │  │ API │
└─────┘  └─────┘
```

講師コメント:
```
MCPサーバーが翻訳者の役割を果たします。
AIが「顧客一覧を見せて」と言うと、
MCPサーバーがDBに問い合わせて結果を返す。

AIは「MCPに対応したサーバーがあれば何でも参照できる」
ということです。
```

## MCPの注意点

### セキュリティ

講師トーク例:
```
MCPは便利ですが、セキュリティには十分注意が必要です。

1. 機密情報の漏洩リスク
   - AIに渡したデータは外部サーバーに送信される可能性
   - Anthropic APIを使う場合、データの取り扱いポリシーを確認

2. アクセス制御
   - MCPサーバーには認証機構を設ける
   - 必要最小限のデータのみ公開

3. ログ管理
   - AIがどのデータにアクセスしたか記録
   - 監査対応のため

社内導入時は、情報セキュリティ部門と連携してください。
```

### パフォーマンス

```
MCPサーバーの応答速度がAIの応答速度に直結します。
大量のドキュメントを検索する場合は:

1. ベクトルDBの活用（Pinecone, Milvusなど）
2. キャッシュ機構の導入
3. 検索結果の件数制限

「速くて正確」が理想ですが、トレードオフを意識してください。
```

## 想定される質問と回答

### Q1: MCPサーバーは自分で作る必要がある？

A: 「汎用的なものは公開されています。」
- ファイルシステム: `@anthropic/mcp-server-filesystem`
- Google Drive: `@anthropic/mcp-server-gdrive`
- GitHub: `@anthropic/mcp-server-github`
- PostgreSQL: `@anthropic/mcp-server-postgres`

「自社専用のデータソースに接続する場合は、カスタムサーバーが必要です。
ただし、仕組みはシンプルなので、数日で作れます。」

### Q2: どのAI製品がMCPに対応している？

A: 「現時点での対応状況です。」
- Claude Code: 完全対応
- Claude Desktop App: 対応
- Cursor: 部分対応（設定次第）
- GitHub Copilot: 直接対応なし（別途連携必要）
- ChatGPT: 非対応（GPTsで類似機能）

「Anthropicが提唱したプロトコルなので、Claude系が強いです。
ただし、オープン規格なので他社も追随する可能性があります。」

### Q3: 社内の全ドキュメントをMCPで公開して大丈夫？

A: 「全公開は推奨しません。」
- 機密レベルで分類し、公開範囲を決める
- 個人情報を含むドキュメントは除外
- アクセスログを取得し、異常検知を行う

「『AIに質問できるドキュメント』と『できないドキュメント』を
明確に分けることが重要です。」

### Q4: ローカルLLMでもMCPは使える？

A: 「理論的には可能ですが、実装状況は様々です。」
- Ollama: 一部のクライアントがMCP対応
- LM Studio: 現時点で非対応
- LocalAI: コミュニティで開発中

「MCP自体はプロトコル仕様なので、クライアント側の対応次第です。
エンタープライズ利用なら、クラウドAPIの方が安定しています。」
